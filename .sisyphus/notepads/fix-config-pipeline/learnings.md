# Learnings — Fix Config Pipeline

## Conventions & Patterns

(Inherited from init-only-theme plan — see `.sisyphus/notepads/init-only-theme/learnings.md`)

### Key Patterns from Previous Work

- State is per-file, options are per-compilation closures
- Plugin-level variables need closure scope (NOT inside Program.enter)
- `addConfigImport()` expects ABSOLUTE path, computes relative automatically
- `findTailwindConfig()` is cached — safe to call repeatedly
- Vitest mocking: `vi.mock()` with `vi.importActual()` for partial mocks
- Test filenames need absolute paths for deterministic relative path computation

### [2026-02-28] Task 5: Documentation Architecture Reversion

**Context**: Commit b5de6d0 (Task 8 of init-only-theme) incorrectly updated docs to match broken implementation (direct provider imports). Task 5 reverts to correct 3-layer architecture.

**Key Findings**:

1. **3-Layer Architecture (Correct)**:
   - Layer 1: User's provider module exports `provideConfig(originalConfig) => config`
   - Layer 2: `.generated.tailwind.config.js` (generated by plugin) — imports provider, calls `provideConfig()`, exports `__twConfig`
   - Layer 3: Components import `__twConfig` from generated file (NOT from user's provider)

2. **Documentation Updates**:
   - `feature-init-only-theme.md`: Updated babel config to show `importName: "provideConfig"`, changed component import to `.generated.tailwind.config.js`, added generated file example
   - `init-only-theme-implementation-plan.md`: Updated Goal section to mention "generated config", added `importName` parameter to scope, updated section 4.5 to describe generated module creation

3. **QA Results**:
   - ✓ Generated file references: 2 in feature proposal, 3 in implementation plan
   - ✓ No "direct provider import" language found in either doc
   - ✓ 3-layer architecture clearly documented with code examples
   - ✓ All acceptance criteria met

4. **Pattern**: Documentation must always reflect the intended architecture, not the broken implementation. When implementation diverges from design, docs should be reverted to design, not updated to match broken code.

### [${TS}] Task 2: Program.enter generation pipeline wiring

- Wired config-module generation immediately after `Object.assign(state, initialState)` so each file has initialized theme/options before generation decisions run.
- Kept `generatedConfigPaths` in plugin factory closure scope to deduplicate writes across all files in a compilation pass.
- Guarded generation with `state.configProviderEnabled`, then null-guarded `findTailwindConfig(...)` with warning and skip semantics, leaving `state.generatedConfigPath` unchanged when no config exists.
- Set `state.generatedConfigPath` to absolute generated module path when config discovery succeeds, enabling downstream visitors to consume a stable resolved path.
